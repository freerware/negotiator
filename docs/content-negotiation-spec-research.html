01Executive summary

negotiator is an admirably ambitious library. It is one of very few Go projects that attempts to implement all three of the content-negotiation strategies described in the HTTP specifications — proactive (server-driven), reactive (agent-driven), and transparent — rather than just parsing an Accept header and calling it a day. The transparent path in particular implements the Remote Variant Selection Algorithm (RVSA/1.0) and the Alternates/TCN/Negotiate header machinery, which almost nothing in the wild bothers to do.

The core algorithms are faithful in spirit. The Apache httpd selection algorithm is reproduced step-for-step, and the RVSA quality computation follows RFC 2296 closely (including the round-to-five-decimals rule that Apache itself skips). The discrepancies that follow are therefore mostly about edges: standards that moved underneath the code, HTTP wire-format details, and a handful of places where a representation is dropped (or a panic is raised) when the specification would have it served or surfaced as an error.

The three headline findings
  • No Vary header is ever emitted. Every negotiated response is cache-poisonable, because downstream caches are never told which request headers drove the choice (RFC 9110 §12.5.5). High
  • A non-existent Content-Charset response header is written on every response. HTTP has no such field; charset travels in the charset parameter of Content-Type. High
  • The library is written against RFC 7231, which has been obsoleted by RFC 9110 (June 2022). Among other things, RFC 9110 deprecates Accept-Charset — a header the library treats as a first-class negotiation dimension with its own strict mode. High

03The specification landscape

“Content negotiation” is not one specification but a small constellation of them, written across two decades and at very different levels of maturity. Understanding the discrepancies requires first understanding that the ground has shifted.

Two of the four core specs are Experimental and effectively dead on the wire

RFC 2295 and RFC 2296 never advanced beyond Experimental and saw almost no client adoption; mainstream browsers do not send Negotiate or understand Alternates/TCN. This does not make the library wrong — implementing them is the whole point — but it frames the transparent findings below as conformance-to-an-experiment rather than conformance-to-the-live-web.

04How the library is built

The design is clean and maps directly onto the specification taxonomy. A single Negotiator interface (negotiator.go) is implemented three times, one per strategy, each constructed with functional options and each delegating the “which variant wins” decision to a pluggable representation.Chooser:

negotiator.Negotiator         interface { Negotiate(ctx, ...Representation) error }
 ├─ proactive.Negotiator       → Chooser = ApacheHTTPD()   (8 ordered filters)
 ├─ reactive.Negotiator        → emits 300 Multiple Choices + list body
 └─ transparent.Negotiator     → Chooser = RVSA1()       (RFC 2296 overall quality)

internal/header/*   parsing & modelling of every Accept-* / Negotiate / Alternates / TCN field
representation/*     the Representation model, the List representation, source-quality, marshalling

The internal/header package is where the specifications are encoded most literally — media ranges, charset/coding/language ranges, quality values, feature lists and predicate bags — and it is where most of the wire-format findings live. The representation package supplies the content (JSON/XML/YAML marshalling, gzip/deflate/compress encoding) and the metadata (media type, charset, language, encoding, features, source quality) that the choosers rank.

05Proactive (server-driven) negotiation

RFC 7231 §3.4.1 / RFC 9110 §12.1–12.5 · Apache httpd 2.4 algorithm · source: proactive/proactive.go, proactive/apache_httpd.go

The proactive flow parses Accept, Accept-Language and Accept-Charset; runs a per-header “strict mode” check that returns 406 when no representation is compatible with a header that is configured strict; and otherwise hands the set to the Apache chooser to pick the best variant (or 406 if none survives).

The Apache algorithm is reproduced faithfully Conformant

The eight filters in apache_httpd.go:39-56 line up one-to-one with the published Apache 2.4 dimensions, with variants.First() standing in for the final “select the first remaining variant” tie-break (Apache step 9):

Where it diverges

The divergences (detailed in the catalogue as D5–D9, D14) cluster around two ideas. First, up-front global elimination: the chooser discards any variant whose quality in any dimension rounds to zero before the ordered filters even run (apache_httpd.go:99-103). Apache instead applies elimination per-step and, crucially, gives a language with no Accept-Language match a tiny non-zero quality rather than zero — so Apache can still serve a language-mismatched variant as a last resort where this library returns 406 (D7).

Second, quality values are read but the “not acceptable” (q=0) meaning is not honoured by strict mode: the strict-mode pre-check calls Accept.Compatible(), which only tests media-range matching and ignores the q value (proactive.go:197 → media_range.go:141). A representation that matches only a q=0 range is counted “compatible” and passes strict mode (D6). Relatedly, Accept-Encoding has no strict mode at all (proactive_options.go), and the identity coding is not treated as always-acceptable the way RFC 9110 §12.5.3 requires (D8).

RFC 9110 §12.4.2: “A sender of qvalue MUST NOT generate more than three digits after the decimal point. … a quality value of 0 means ‘not acceptable’.”

06Reactive (agent-driven) negotiation

RFC 7231 §3.4.2 / RFC 9110 §12.2 & §15.4.1 · source: reactive/reactive.go

The reactive negotiator is intentionally thin: it serializes a List representation describing every available variant and returns it with 300 Multiple Choices. That is the canonical agent-driven behaviour — the server hands the client the menu and lets the client choose — and the strategy is essentially conformant.

The gaps are minor (D10–D11): the 300 response carries no Location header pointing at a preferred default (RFC 9110 §15.4.1 says it SHOULD when one exists), no Vary, and the metrics double-count — the deferred counter increments negotiate.multiple_choices on every non-error return, including the empty-set path that has already incremented negotiate.no_content and written 204 (reactive.go:96-110).

07Transparent negotiation

RFC 2295 (protocol) + RFC 2296 (RVSA/1.0) · source: transparent/transparent.go, transparent/rvsa.go, internal/header/{negotiate,alternates,tcn,accept_features,feature_*}.go

This is the most impressive — and the most exposed — part of the library. It reads the Negotiate request header, decides whether the user agent wants the server to choose (*, 1.0, or guess-small), runs RVSA/1.0 to compute an overall quality per variant, and then either returns a choice response (when a single “definite” best variant exists and is a URL neighbour of the negotiable resource) or a list response carrying Alternates and TCN headers.

What it gets right Conformant

  • Overall-quality computation — qs × qt × qc × ql × qf, rounded to five decimal places (rvsa.go:215-224), matching RFC 2296 §3.3 exactly (and notably more faithfully than Apache, which skips the rounding).
  • Definiteness gate — a choice response is only produced when no dimension relied on a wildcard or an absent request header (IsDefinite, rvsa.go:84,111), the correct reading of “the server may only return a choice it is certain about.”
  • Neighbour test — the chosen variant’s Content-Location must share the negotiable resource’s parent path (RFC 2296 §3.5), with RFC 2068 URL equivalence (transparent.go:182-237).
  • guess-small — falls back to a list when the choice body is not smaller than, and is more than the threshold larger than, the list body (transparent.go:158-179).

Where it diverges

The notable issues (D12, D15–D16, D18–D19) are wire-format and robustness:

  • Malformed Alternates attributes (D12). NewAlternates emits every variant attribute unconditionally, so a variant with no charset/language/features yields empty tokens like { charset  } and { language  } (alternates.go:113-122). RFC 2295 §8.3 makes each variant-attribute optional; it should be omitted when empty, not emitted blank.
  • A parse error becomes a panic (D15). If a variant’s feature list fails to parse during selection, RVSA calls panic(err) (rvsa.go:205-208) instead of returning the error the Chooser signature already provides — a single bad variant takes down the request handler.
  • Encoding dropped from RVSA (D18). The Accept-Encoding dimension is stubbed out (rvsa.go:42-59). RVSA/1.0 itself does not define an encoding factor, so this is defensible, but it means transparent selection ignores encoding entirely.
  • Feature quality is unbounded (D16). QualityDegradation multiplies true-improvement factors that may exceed 1.0 and feeds the raw product straight into header.QualityValue, bypassing the 0.0–1.0 validation in NewQualityValue (feature_list.go:47-57, rvsa.go:209-212), so overall quality can rise above 1.0.
  • No 506 Variant Also Negotiates (D17) — the configuration-error status from RFC 2295 §8.1 is never produced.

08Cross-cutting issues

Three problems are independent of strategy because they touch every response the library writes.

No Vary header, anywhere

For proactive (and indeed transparent) negotiation, RFC 9110 §12.5.5 states the server SHOULD generate a Vary field listing the request-header fields that influenced selection, so that shared caches key their stored responses correctly. The library sets Content-* fields but never Vary (proactive.go:270-275, 344-348; reactive.go:130-134; transparent.go:275-281, 338-345). A cache in front of an application using this library will happily serve a French, gzip’d variant to a client that asked for English plaintext.

The phantom Content-Charset header

Every response path writes ctx.ResponseWriter.Header().Add("Content-Charset", cc). There is no Content-Charset field in HTTP. The selected charset must be conveyed as the charset parameter of Content-Type (e.g. Content-Type: application/json; charset=utf-8). Clients and caches ignore the bogus field, and the actually-negotiated charset is therefore never communicated.

Built against an obsoleted RFC

The code, comments, and README cite RFC 7231 and link to tools.ietf.org. RFC 7231 was obsoleted by RFC 9110 in June 2022. The negotiation model is largely unchanged, but RFC 9110 §12.5.2 now marks Accept-Charset as deprecated (“servers ought to ignore it”; modern stacks assume UTF-8). The library elevates Accept-Charset to a first-class dimension with its own strict mode — investing conformance effort in a header the current standard is retiring.
